Git & GitHub Repo Creation

A complete terminal guide — from zero to a live GitHub repository, explained step by step.

📋 All Commands — Quick Reference

# 1. Check if a git repo already exists
git status

# 2. Initialize a new git repository
git init

# 3. Stage all files for tracking
git add .

# 4. Commit staged files with a message
git commit -m "Initial commit"

# 5. Authenticate with GitHub (one-time)
gh auth login

# 6. Create the repo on GitHub & link it
gh repo create your-repo-name --public --source=.

# 7. Push your code to GitHub
git push -u origin master
1

Check Repository Status

git status

Before doing anything, run this command to check whether your folder is already a Git repository.

  • git — the Git program installed on your computer. It's the version control system that tracks every change you make to your files.
  • status — asks Git: "What's the current state of this folder?" It shows which files are tracked, modified, or staged.
💡 What to expect: If you see "fatal: not a git repository", it means Git hasn't been set up yet in this folder — which is exactly why we run the next step.
2

Initialize a Git Repository

git init

This creates a brand-new, empty Git repository in your current folder.

  • git — calls the Git program.
  • init — short for initialize. It creates a hidden .git folder inside your project directory. This folder is where Git stores all its internal data — history, branches, configs, etc.
💡 Good to know: You only need to run this once per project. After this, Git will quietly track everything inside this folder.
3

Stage All Files

git add .

This tells Git which files you want to include in the next save (commit). Think of it as putting items into a box before sealing it.

  • git — calls the Git program.
  • add — tells Git to stage files, meaning "prepare these files for the next commit."
  • . — the dot means everything in the current directory and all subdirectories. It stages all new and modified files at once.
💡 Alternatively: You can stage individual files with git add filename.html if you don't want to include everything.
4

Commit Your Files

git commit -m "Initial commit"

This saves a snapshot of all your staged files into Git's history. It's like taking a photo of your project at this exact moment.

  • git — calls the Git program.
  • commit — creates a permanent snapshot (a "save point") of everything you staged with git add.
  • -m — stands for message. This flag lets you write a short description directly in the command, without opening a text editor.
  • "Initial commit" — the message itself. It describes what this snapshot contains. You can write anything here, e.g. "Added homepage" or "Fixed login bug".
⚠️ Important: The message must be in quotes. Without the -m flag, Git opens a text editor (usually Vim), which can be confusing for beginners.
5

Authenticate with GitHub

gh auth login

Before you can create repos on GitHub from the terminal, you need to prove who you are. This command starts the login process.

  • gh — the GitHub CLI (Command Line Interface). It's a separate tool from Git that lets you interact with GitHub directly from the terminal — create repos, open pull requests, manage issues, etc.
  • auth — the authentication module. Handles everything related to logging in and out.
  • login — starts an interactive login flow. It will ask you to choose GitHub.com, select HTTPS, and authenticate via a web browser.
💡 One-time only: You only need to log in once. After this, the GitHub CLI remembers your credentials. To check your login status later, run gh auth status.
6

Create the GitHub Repo & Link It

gh repo create your-repo-name --public --source=.

This creates a new repository on GitHub and links it to your local folder — but doesn't upload files yet.

  • gh — calls the GitHub CLI tool (separate from Git itself).
  • repo — the repository module. Manages all repo-related operations (create, delete, clone, fork, etc.).
  • create — tells GitHub to create a brand‑new repository on their servers.
  • your-repo-name — the name of the repository on GitHub. Replace this with whatever you want to call it (e.g. my-portfolio, www1, cool-project).
  • --public — makes the repo visible to everyone. Use --private instead if you want only you to see it.
  • --source=. — tells the GitHub CLI that the source code is in your current directory (the dot . means "here"). This also automatically sets up the remote link (called "origin") between your local folder and the GitHub repo.
💡 What happens: After this, GitHub creates the repo and your local folder knows where to send code. But the files aren't uploaded yet — that's the next step.
7

Push Your Code to GitHub

git push -u origin master

This uploads your committed files from your computer to GitHub. After this, your code is live!

  • git — calls the Git program.
  • push — sends your local commits to a remote server (in this case, GitHub).
  • -u — stands for upstream. It tells Git to remember this connection so that next time, you can just type git push without specifying where.
  • origin — the default name for the remote GitHub repository. This was automatically set up by the previous step (--source=.).
  • master — the name of your branch. This is the default main branch that was created when you ran git init.
💡 Result: Your code is now live on GitHub! Visit https://github.com/yourusername/your-repo-name to see it. From now on, just use git push to upload future changes.

🔄 After Setup — Everyday Commands

Once your repo exists, use these 3 commands whenever you make changes:

# Stage all changed files
git add .

# Save a snapshot with a description
git commit -m "describe your changes"

# Upload to GitHub
git push
